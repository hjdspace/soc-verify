# Diff Review System — 全局队列 + 延迟批量应用 + 精确撤销

AI Agent 的 WRITE/EDIT 工具在执行时已将改动写入磁盘。我们选择不在 AI 工具层阻止写入（不改为暂存模式），而是在 UI 层增加 Diff Review 流程——用户可以在中栏 `DiffReviewView` 中审阅完整文件 diff，逐 hunk 接受/拒绝，延迟批量应用。

## 关键决策

1. **全局 Review Queue**：跨所有会话汇总改动，而非绑定到单个会话。跨会话编辑同一文件时按时间逆序撤销重建 before 状态。

2. **延迟批量应用**：用户在 UI 中逐个勾选 hunk 状态，「应用」时后端一次性执行所有拒绝操作。不采用即时逐次写磁盘。

3. **精确撤销**：后端基于 toolArgs 中的 oldText/newText 进行精确替换撤销，不依赖 git。EDIT 用 oldText 替换 newText；WRITE 删除新文件。

4. **Diff 计算在后端**：新增 `project.getFileDiff` API，后端读当前文件、重构 before、计算结构化 diff。前端只负责渲染。

## 被拒绝的方案

- **暂存模式**（AI 不直接写磁盘，接受后才写入）：需改造所有 AI 工具和 omp 引擎，改动面过大且违背「不修改 omp 引擎源码」约束。
- **git 恢复**：项目不一定在 git 管理下，且 git 恢复粒度为文件级，无法做 hunk 级精确撤销。
- **即时应用**：每次点击立即写磁盘，用户改主意需再次操作，且频繁 IO 影响体验。
- **按会话队列**：多会话场景下用户需切换会话才能审阅不同改动，不符合其他 AI IDE 的交互习惯。
